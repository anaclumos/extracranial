[Oneday](./../.././docs/pages/Oneday.md) [Project](./../.././docs/pages/Project.md).

The new [ArXiv](./../.././docs/pages/ArXiv.md).

[ArXiv](./../.././docs/pages/ArXiv.md) is excellent, but it by no means matches the full-on productivity of modern DevOps. We need a more streamlined, so-called **ResearchOps**--a [Super App](./../.././docs/pages/Super%20App.md) platform for all research.

TeX sucks (or at least we need better [interoperability](./../.././docs/pages/Interoperability.md) with HyperText Media), and [PDF](./../.././docs/pages/PDF.md) sucks. Citing by their name sucks. It's like managing PRs by [email](./../.././docs/pages/Mail.md). Remember Linus used to do that?

The current day [ArXiv](./../.././docs/pages/ArXiv.md) is like

- SourceForge, not [GitHub](./../.././docs/pages/GitHub.md)
- Kaggle, not HuggingFace

And people only know what they want after seeing the future.

- What if we add interactive widgets in the middle of the papers?
- What if we add [3D](./../.././docs/pages/3D.md) models right in the papers?
- Citations are automatically generated, similar to how package.json works in the npm install
- What if we can auto-translate to all locales? [ACL 60-60](./../.././docs/pages/ACL%2060-60.md)
- and so on...
